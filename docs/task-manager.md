# Task Manager v1

Task Manager v1 is the first real internal work-state layer in ASSEM.

It does not implement advanced planning yet. Its job in this phase is simpler and stricter:

- keep a persisted task record per session
- expose real status and progress data
- let the orchestrator answer task questions without inventing state

## What It Stores

Each task stores at least:

- `id`
- `sessionId`
- `objective`
- `status`
- `progressPercent`
- `currentPhase`
- `steps`
- `currentStepId`
- `artifacts`
- `createdAt`
- `updatedAt`
- `startedAt`
- `completedAt`
- `pausedAt`
- `failureReason`
- `metadata`

In this phase `metadata` can also hold interrupt-related state for the active task, such as:

- persisted refinements
- last interruption timestamp
- last clarification message

Task records can now also store:

- a persisted `TaskPlan`

## Supported States

- `pending`
- `active`
- `paused`
- `blocked`
- `completed`
- `failed`
- `cancelled`

## Persistence

Task Manager persists locally in:

- `ASSEM_DATA_ROOT/tasks.json`

It is intentionally separated from:

- `sessions.json`
- `profiles.json`
- `scheduler.json`
- `telemetry.jsonl`

That means:

- conversation history stays in session storage
- profile memory stays in the memory backend
- scheduler state stays in the scheduler store
- task progress stays in the Task Manager store

## Orchestrator Integration

In this phase the orchestrator reads Task Manager directly for:

- `que estas haciendo`
- `cuanto te queda`
- `en que vas`
- `pausa`
- `reanuda`
- `cancela`

When there is an active task, Interrupt Handler v1 also uses Task Manager-backed state to decide whether the latest message is:

- a task status query
- a task control action
- a safe refinement
- or an independent query that should continue through the normal chat/tool flow

Important behavior:

- if there is no active task in the current session, ASSEM says so clearly
- if there is progress data, ASSEM answers from that value
- if there is no reliable percentage yet, ASSEM says that honestly
- task phase and current step are read from Task Manager state, not inferred by the model

Task Runtime / Executor v1 is the layer that now writes those state transitions for runtime-backed tasks.

Planner v1 is the layer that now creates the initial task structure before runtime execution begins.

## Telemetry

Task Manager emits local telemetry events for:

- `task_created`
- `task_started`
- `task_progress_updated`
- `task_paused`
- `task_resumed`
- `task_completed`
- `task_failed`
- `task_cancelled`

These events are stored separately from action history.

## What This Phase Does Not Do Yet

- multi-step planning generated by the model
- multiple simultaneous active tasks in one session
- autonomous background execution loops
- worker/sub-agent orchestration
- browser automation
- desktop automation
- artifact generation pipelines

This phase is the persistence and truth layer that later planning work can build on top of.
